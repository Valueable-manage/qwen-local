# RAG 流程分析

## 一、整体架构

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  数据流                                                                         │
│                                                                                 │
│  docx 文档 → parse_docx → 翻译(可选) → build_index → ChromaDB + BM25            │
│                                                                                 │
│  用户提问 → 查询改写 → 向量检索 + BM25 检索 → 合并去重 → 重排序 → 过滤 → top-3   │
│                                                                                 │
│  top-3 上下文 + 用户问题 → Qwen 生成回答                                         │
└─────────────────────────────────────────────────────────────────────────────────┘
```

---

## 二、核心模块（rag_pipeline.py）

### 1. 文档解析 `parse_docx(path)`

- **输入**：`.docx` 文件路径
- **输出**：`[{"id", "问题", "答复", "图号"}, ...]`
- **逻辑**：按「问题 + 答复」配对提取，识别 `答复：` 作为分隔
- **领域**：针对船舶工程「疑难意见处理」文档格式（含 DE24/64/249 等图号）

### 2. 索引构建 `build_index(entries)`

| 组件 | 用途 | 存储 |
|------|------|------|
| **ChromaDB** | 向量检索（语义相似） | `rag_db/chroma/` |
| **BM25** | 关键词检索 | `rag_db/bm25.pkl` |
| **entries** | 原始条目缓存 | `rag_db/entries.json` |

- **嵌入模型**：`BAAI/bge-small-zh-v1.5`（约 100MB）
- **索引文本**：`问题 + 答复` 拼接
- **BM25 分词**：jieba 中文分词

### 3. 检索流程 `search(query, top_k=3)`

```
1. 查询改写（可选）
   └── 用 Qwen 把口语问题扩展为专业术语，提升召回率

2. 双路召回
   ├── 向量检索：ChromaDB cosine similarity，取 top_k*2
   └── BM25 检索：关键词匹配，取 top_k*2

3. 合并去重
   └── 按「问题」前 50 字去重，向量结果优先

4. 重排序
   └── Cross-encoder（BAAI/bge-reranker-base，约 280MB）精排

5. 过滤 + 截断
   ├── 相似度 < 0.3 丢弃
   └── 取 top_k
```

### 4. 初始化 `init_rag(docx_path)`

- **中文版**：若文件名为 `*_中文版.docx`，直接解析，**跳过翻译**
- **翻译**：否则若 `*_translated.json` 不存在，用 Qwen 将英文翻译为中文
- **索引**：构建 ChromaDB + BM25

**文档格式**：支持 `.docx` 和 `.doc`；按标号（1、2、3…）切割为 chunk。
- **输出**：`*_整理版.docx`（可选，翻译后结构化文档）

## 三、与 Web 的集成（server.py）

### 1. 启动时

```
lifespan
  ├── load_model()
  ├── _init_all_rag()
  │   └── 扫描 rag_docs/*.docx，逐个 init_rag
  └── _rag_ready = True（任一成功）
```

### 2. 对话时

```
/v1/chat/completions
  ├── 若 _rag_ready
  │   ├── 取最后一条 user 消息
  │   ├── search(query, top_k=3)
  │   └── 将 top-3 注入为：参考1/2/3 + 问题
  └── 调用 chat/chat_stream 生成回答
```

### 3. 文档管理

| 接口 | 作用 |
|------|------|
| `POST /api/upload-doc` | 上传 docx；保存到 rag_docs；后台线程建索引 |
| `GET /api/doc-status` | 返回 rag_ready、docs 列表 |
| `POST /api/clear-doc` | 删除 rag_docs、rag_db，重置 _rag_ready |

```
用户输入："VECS管子要直"
    │
    ▼
查询改写
    "VECS管子要直 VECS drain pipe 排放管 直线 净空 clearance upper deck"
    │
    ▼
向量检索
    ChromaDB 返回 6 条候选
    │
    ▼
BM25 检索
    返回 6 条候选
    │
    ▼
合并去重
    │
    ▼
Cross-encoder 重排序
    │
    ▼
过滤 + top-3
    │
    ▼
注入 prompt
    "以下是相关的船舶工程处理意见，请根据这些内容回答问题。
     【参考1】问题：... 答复：... 图号：...
     【参考2】...
     【参考3】...
     问题：VECS管子要直"
    │
    ▼
Qwen 生成回答
```

## 六、配置要点

| 配置 | 位置 | 说明 |
|------|------|------|
| EMBED_MODEL | rag_pipeline | 嵌入模型 |
| RERANK_MODEL | rag_pipeline | 重排序模型 |
| SIMILARITY_THRESHOLD | rag_pipeline | 0.3，低于此分数丢弃 |
| top_k | search() | 默认 3，注入上下文条数 |
| RAG_DOCS_DIR | server | rag_docs/，文档存放目录 |

---

## 七、注意事项

1. **首次加载**：需下载 bge-small-zh、bge-reranker-base，约 380MB
2. **首次建索引**：若需翻译，会逐条调用 Qwen，较慢
3. **文档格式**：依赖「问题 + 答复：」结构，非通用 docx
4. **多文档**：rag_docs 下多个 docx 会依次 init。ChromaDB 会追加；BM25 每次 build_index 会重建，当前实现下只保留**最后一个**文档的 BM25 索引（潜在问题）
